ERROR: CucumberException - Zero-Argument Constructor
════════════════════════════════════════════════════════════════════════════

FULL ERROR MESSAGE:
───────────────────
io.cucumber.core.exception.CucumberException: class com.qa.hooks.Hooks does 
not have a public zero-argument constructor.

To use dependency injection add an other ObjectFactory implementation such as:
 * cucumber-picocontainer
 * cucumber-spring
 * cucumber-jakarta-cdi
 * ...etc

Caused by: java.lang.NoSuchMethodException: com.qa.hooks.Hooks.<init>()


WHY THIS HAPPENS:
─────────────────

Your Hooks and StepDefinition classes use Constructor Dependency Injection:

  public Hooks(ScenarioContext scenarioContext) {
      this.scenarioContext = scenarioContext;
  }

Cucumber needs a dependency injection framework to handle this.
By default, Cucumber uses DefaultObjectFactory which requires a zero-argument
constructor (public Hooks() {}).

Since we have a constructor with parameters, Cucumber cannot instantiate the
class without a DI framework.


THE FIX:
────────

Add cucumber-picocontainer dependency to pom.xml:

  <dependency>
      <groupId>io.cucumber</groupId>
      <artifactId>cucumber-picocontainer</artifactId>
      <version>${cucumber.version}</version>
  </dependency>

✅ This has already been added to your pom.xml
✅ Your pom.xml now has the correct dependency

WHAT CUCUMBER-PICOCONTAINER DOES:
──────────────────────────────────

When you add this dependency, Cucumber will:

1. Automatically detect classes with @Before, @Given, @When, @Then, @After
   annotations

2. Look at their constructors and parameters

3. For each parameter (e.g., ScenarioContext):
   ├─ Check if it has a @Before/@After hook as well
   ├─ If yes, create a single shared instance
   ├─ Inject the same instance into all classes that need it

4. This enables:
   ├─ Hooks sets data: scenarioContext.setTestData(data)
   ├─ Steps reads data: scenarioContext.getTestData()
   └─ Both use the SAME instance


EXAMPLE DEPENDENCY INJECTION FLOW:
──────────────────────────────────

Cucumber discovers:
  ├─ Hooks class (has @Before)
  ├─ LoginPageStepDefs class (has @When, @Then)
  └─ ScenarioContext class (no annotations, but needed by both)

Cucumber PicoContainer:
  ├─ Creates one instance of ScenarioContext
  ├─ Injects into Hooks constructor: new Hooks(scenarioContext)
  ├─ Injects into StepDefs constructor: new LoginPageStepDefs(scenarioContext)
  └─ Both classes share the SAME scenarioContext instance

Result:
  ✓ Hooks can set data in context
  ✓ Steps can read data from context
  ✓ Data persists between hooks and steps
  ✓ Scenario iteration data is shared correctly


ALTERNATIVE SOLUTIONS (NOT RECOMMENDED):
─────────────────────────────────────────

If you can't use cucumber-picocontainer, you could:

Option 1: Add Zero-Argument Constructor
  ❌ Not recommended - violates dependency injection pattern
  └─ public Hooks() { } + parameterized constructor
  └─ Problem: ScenarioContext won't be injected

Option 2: Use Static Context
  ❌ Not recommended - causes issues with parallel execution
  └─ public static ScenarioContext context = new ScenarioContext();
  └─ Problem: Not thread-safe, can't run tests in parallel

Option 3: Use cucumber-spring or cucumber-jakarta-cdi
  ⚠️ More complex setup - only if you need Spring/Jakarta CDI
  └─ Good if your project already uses Spring/Jakarta
  └─ For BDD Playwright tests, cucumber-picocontainer is simpler

✅ RECOMMENDED: Use cucumber-picocontainer (already done in your pom.xml)
